Step 1 — Codility Test (Online)

⏰ 60–120 min 🎯 Algorithms & Code Reading

Goal: Quickly verify your algorithmic foundations and ability to read/fix existing code.

Typical format: 2–3 exercises, sometimes "fix existing buggy code".

Actual questions reported (Nov 2025):

Problem 1 — Nails & Hammer

Review code and make max 4 fixes. Hammer down at most K nails — maximize the number of nails that end up having the same length.

Problem 2 — Remove Adjacent Pairs

Pairs are a/b and c/d. Example: CBACD → CBA → C. Stack-based approach.

Problem 3 — Binary Reduction to Zero

If v is odd → subtract 1. If even → divide by 2. Count operations.

Example: 011100 (28) → 11100 (28) → 14 → 7 → 6 → 3 → 2 → 1 → 0 = 7 ops

🔍 What They Evaluate
  • Quick understanding of the problem statement
  • Clean code, edge case testing
  • Reasonable complexity (not naive if it blows up, but not necessarily ultra-optimal)
📚 Preparation
  • Do 20–30 Codility/LeetCode exercises (strings, stack, two pointers, binary)
  • Practice reading buggy code and fixing it fast (minimal refactor)
❓ Questions to Ask (if applicable)

Usually no interaction during Codility, but if there's a preamble:

  • "Is partial credit given for passing some test cases?"
  • "Should I optimize for readability or performance?"

Step 2 — Initial Video Interview (HR / Recruiter)

⏰ 30 min 🎯 Fit + Motivation + Logistics

Goal: Filter for culture fit, motivation, and logistics.

Content:

  • Your background (2–3 minutes)
  • Why Sonar / why Geneva / why this role
  • Salary expectations, availability, visa, languages
🔍 What They Evaluate
  • A clear story — what you do, what you want, why them
  • Professional attitude, simple communication
  • Enthusiasm for the product and mission
📚 Preparation
  • 60-second pitch + 2-minute version
  • 3 concrete reasons "for Sonar": code quality, dev tooling, product impact, engineering culture
  • Know your salary range, availability, visa status
❓ Questions to Ask HR
  • "What does the team structure look like for this role?"
  • "What's the interview timeline after this call?"
  • "What does Sonar value most in its engineering culture?"
  • "Is this a new role or backfill?"

Step 3 — Technical Interview (Quick Screen)

⏰ 30 min 🎯 Engineering Mindset Validation

Goal: Express technical validation before investing more time. They want to see how you think, how you explain, and your Java/API/design basics.

🔍 What They Evaluate
  • Clear reasoning — structured thought process
  • Java best practices — naming, patterns, OOP
  • Ability to explain choices — "why" not just "what"
Example 1 — Quick Java Refactoring

They give you:

❌ Bad Code (Given)
public void process(User u){
   if(u != null){
      if(u.getStatus() == 1){
         if(u.getEmail()!=null){
            sendMail(u.getEmail());
         }
      }
   }
}
✅ Senior Answer — Guard Clauses
public void process(User user){
   if(user == null) return;
   if(!user.isActive()) return;

   String email = user.getEmail();
   if(email == null) return;

   sendMail(email);
}

Explain: fewer nested conditionals, more readable, more testable.

Example 2 — API Question

They ask: "Would you send data in a GET request?"

✅ Senior Answer

GET requests should not contain a body according to REST conventions.

They should be idempotent and cacheable.

  • GET = read only
  • POST = create
  • PUT = replace
  • PATCH = partial update
  • DELETE = remove
Example 3 — Scalability

They ask: "What happens if 10,000 users call your API simultaneously?"

✅ Senior Answer
  • Thread pool management
  • Async processing
  • Caching with Redis
  • Load balancing

"I would first cache frequent reads, then ensure horizontal scaling behind a load balancer."

📚 Preparation
  • Practice explaining out loud: "assumptions → approach → complexity → edge cases → tests"
⚠️ Warning

In the reported experience, the candidate felt confident but was told "not moving forward" at the end. This can happen if:

  • Different expectations (level, stack, design)
  • Communication not structured enough
  • Internal decision (budget, role frozen, etc.)
❓ Questions to Ask the Interviewer
  • "What does a typical day look like for this role?"
  • "What's the tech stack for the team I'd be joining?"
  • "How do you approach code reviews?"

Step 4 — 2-Hour Technical Interview (The Real Battle)

⏰ 2 hours 🎯 Live Coding + Design + Scalability ⚡ Hardest Stage

Goal: See you work like on the job.

🔍 What They Evaluate
  • Code quality — readable, testable, clean
  • Pragmatic design — not "over-engineering"
  • API/web basics — HTTP, idempotence, caching headers
  • Communication during exercises — talk while coding
Example 1 — Complete Refactoring (Strategy Pattern)
❌ Code Given
if(order.type.equals("VIP")){
   discount = price * 0.20;
}
else if(order.type.equals("NORMAL")){
   discount = price * 0.10;
}
✅ Senior Proposes — Strategy Pattern
interface DiscountStrategy {
   double compute(double price);
}

class VipDiscount implements DiscountStrategy {
   public double compute(double price){
      return price * 0.20;
   }
}

class NormalDiscount implements DiscountStrategy {
   public double compute(double price){
      return price * 0.10;
   }
}

Why? Extensible. Follows Open/Closed Principle. They love this.

Example 2 — API Design

Question: "Design endpoint to upload student courses."

✅ Senior Answer
POST /students/{id}/courses

Body:
{
   "courseId": 10
}

Response codes:
  201 Created    → success
  400 Bad Request → validation error
  404 Not Found   → student not found
  409 Conflict    → already enrolled

Explain: validation, proper HTTP status codes, RESTful naming.

Example 3 — Nested Conditionals

They give deeply nested code:

✅ Senior Techniques
  • Early return / guard clauses
  • Extract methods for clarity
  • Polymorphism when types diverge
  • Strategy / Command pattern for complex branching
Example 4 — Scalability Architecture

Question: "Millions of requests?"

✅ Senior Architecture
Client → API Gateway → Service → Cache (Redis) → DB

Key patterns:
  • Pagination (offset / cursor-based)
  • Rate limiting (token bucket)
  • Circuit breaker (Resilience4j)
  • Horizontal scaling + load balancer
  • Async messaging (Kafka / RabbitMQ)
Example 5 — Talk While Coding

They evaluate your thought process — always verbalize:

✅ The Framework
  1. "First I clarify assumptions."
  2. "Then I design a simple solution."
  3. "Then I optimize."
  4. "Then I discuss edge cases and tests."

This is extremely important. Silent coding = red flag.

Reported technical topics:

  • Refactoring Java: encapsulation, structure, alternative designs
  • API usage: which HTTP methods, sending data in GET requests
  • Scalability: handling multiple requests, caching, load balancing
  • Readability: reducing nested conditionals, naming, early returns, patterns
  • Explaining your approach, thought process, and code decisions throughout
📚 Preparation
  • Do guided refactoring "before/after" exercises
  • Review: GET/POST/PUT/PATCH/DELETE, idempotence, pagination, rate limiting
  • Readability patterns: guard clauses, method extraction, strategy/polymorphism
❓ Questions to Ask the Tech Interviewer
  • "How do you manage technical debt in the codebase?"
  • "What's your CI/CD pipeline like?"
  • "What testing strategy do you follow (unit, integration, E2E)?"
  • "How do you handle breaking changes in the API?"
  • "What's the biggest technical challenge the team is facing right now?"

Step 5 — Cross-functional Interview

⏰ 30 min 🎯 Collaboration — PM / QA / UX

Goal: Are you collaborative or arrogant? Can you work with non-technical people?

🔍 What They Evaluate
  • Explain technical topics to non-tech people
  • Manage trade-offs (quality vs. deadline)
  • Prioritize and clarify requirements
Example Q1 — Product wants quick feature but tech debt risk
✅ Senior Answer

"I explain risks clearly and propose a minimal safe solution."

Example: "We could release MVP behind a feature flag."

They love this — pragmatic + safe.

Example Q2 — QA finds bug before release
✅ Senior Answer
  1. Thank QA — show appreciation
  2. Analyze root cause — not just fix the symptom
  3. Add regression tests — prevent recurrence
Example Q3 — Non-technical stakeholder asks "why it takes so long"
✅ Senior Answer

"I translate technical complexity into business language."

Example: "This feature touches authentication and data integrity — two critical systems that need careful changes."

📚 Preparation
  • Prepare an example: "I simplified a technical decision for non-tech stakeholders"
  • Use a mini-framework: Context → Options → Trade-offs → Decision → Result
❓ Questions to Ask PM / QA / Cross-functional
  • "How do engineering and product collaborate on priorities?"
  • "How are bugs triaged — who decides severity?"
  • "How do you handle scope changes mid-sprint?"
  • "What's the feedback loop between QA and engineering?"

Step 6 — Final Interview with Engineering Leader

⏰ 20 min 🎯 Culture + Seniority + Potential

Goal: Final decision — level, impact, attitude, potential.

🔍 What They Evaluate
  • Seniority level — do you match the expected level?
  • Leadership — influence, mentoring, ownership
  • Mutual expectations — culture, projection, alignment
Example Q1 — "Why Sonar?"
✅ Senior Answer

"I appreciate an engineering culture focused on code quality and developer experience. Sonar's mission to help developers write cleaner, safer code aligns perfectly with how I approach software engineering."

Example Q2 — "Biggest mistake?"
✅ Senior Answer

"I once deployed a change that affected performance in production. I rolled back immediately, set up monitoring alerts, and wrote a post-mortem to prevent recurrence."

Shows: ownership, rapid response, learning.

Example Q3 — "How do you mentor juniors?"
✅ Senior Answer
  • Pair programming sessions
  • Thorough code reviews with teaching comments
  • Creating internal documentation / wiki
  • Setting up learning paths
Example Q4 — "First 90 days plan?"
✅ Senior Answer
  1. Days 1–30: Understand the architecture, read the codebase, onboard properly
  2. Days 30–60: Gain trust, deliver small improvements, build relationships
  3. Days 60–90: Start contributing to bigger features, identify improvement areas
📚 Preparation
  • 2 strong stories (measurable impact, quality/perf improvement, incident managed)
  • Your "senior questions": roadmap, code review standards, tech debt, onboarding, 90-day expectations
❓ Questions to Ask the Engineering Leader
  • "What's the engineering roadmap for the next 6 months?"
  • "What are the code review standards on the team?"
  • "How do you handle technical debt?"
  • "What does the onboarding process look like?"
  • "What would success look like in the first 90 days?"
  • "What's the team's biggest challenge right now?"

Behavioral / Situational Questions (STAR Method)

They want: responsibility + communication + learning.

Use STAR: Situation → Task → Action → Result

Q1 — "A colleague raised a concern about your work"

They want to see: listening + adjustment + team alignment

✅ STAR Answer

S: During a code review, a teammate flagged my database query approach as potentially problematic at scale.

T: I needed to evaluate if my approach was indeed risky.

A: I set up a meeting, listened actively, ran performance benchmarks together, and we agreed on an indexed query with pagination.

R: Query time dropped from 3s to 200ms. We turned this into a team guideline.

Shows: listening, collaboration, team alignment.

Q2 — "You introduced a bug — how did you handle it?"

They want to see: ownership + methodical debugging + prevention

✅ STAR Answer

S: A deployment I pushed caused incorrect calculations in the billing module.

T: Fix the issue, minimize impact, prevent recurrence.

A: I immediately rolled back, traced the root cause (edge case in currency conversion), added regression tests and monitoring alerts.

R: Resolved in 45 min. Zero customer impact due to rollback. Added automated checks in CI pipeline.

Shows: ownership, methodical debugging, prevention.

Q3 — "A challenging situation and how you managed it"

They want to see: stress management + prioritization + communication

✅ STAR Answer

S: Mid-sprint, requirements changed drastically — the client wanted real-time notifications instead of batch emails.

T: Deliver the feature on time with the new requirements.

A: I proposed WebSocket integration with a fallback to polling, broke it into sub-tasks, communicated the trade-offs to PM, and pair-programmed the critical parts with a teammate.

R: Delivered on time. Feature became the most-used by clients. PM praised the transparent communication.

Shows: stress management, prioritization, communication.

📅 7-Day Preparation Plan

A focused 7-day plan covering Codility, refactoring, API/scalability, behavioral, and final polish.

Day 1 — Codility Warm-up
  • 5 easy LeetCode (strings, arrays)
  • 2 medium (stack, two pointers)
  • Practice fixing buggy code snippets
  • Review Big-O basics
Day 2 — Codility Deep
  • 3 medium LeetCode (binary, bitwise)
  • Simulate timed Codility test (90 min, 3 problems)
  • Practice "Nails & Hammer" type problems
  • Review stack-based pair removal
Day 3 — Java Refactoring
  • Guard clauses practice (5 examples)
  • Strategy Pattern implementation
  • Open/Closed Principle exercises
  • Extract method refactoring
  • Before/after code comparisons
Day 4 — API & REST
  • HTTP methods deep review
  • Idempotence & caching headers
  • Design 3 RESTful endpoints from scratch
  • Status codes: 200, 201, 400, 401, 403, 404, 409, 500
  • Pagination & rate limiting concepts
Day 5 — Scalability & Architecture
  • Draw: Client → Gateway → Service → Cache → DB
  • Redis caching patterns
  • Load balancing strategies
  • Circuit breaker (Resilience4j)
  • Kafka / async messaging basics
Day 6 — Behavioral & Cross-functional
  • Write 3 STAR stories (colleague concern, bug intro, challenge)
  • Practice "explain to non-tech" exercise
  • Prepare: feature flag, QA bug, PM trade-off answers
  • First 90 days plan
  • "Why Sonar?" answer polished
Day 7 — Final Polish
  • 60-second & 2-minute pitch out loud
  • Mock interview: refactor + explain (30 min)
  • Review all questions to ask at each stage
  • Quick Codility timed run (3 problems)
  • Rest, sleep well, be confident
💡 Daily Habit (All 7 Days)
  • Talk out loud while solving any problem (practice verbalizing)
  • Always think: "assumptions → approach → complexity → edge cases → tests"
  • Connect every answer back to clean code and quality (Sonar DNA)

  • 🎯 Quality Mindset — tests, reviews, standards

  • 📈 Impact — measurable improvements, not just features

At every stage, connect your answers back to code quality, clean code, and developer experience — that's what Sonar is about.
